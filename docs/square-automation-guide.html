How Square hooks into the framework

When PBI=SQNS, globalSetup.js registers Square handlers into the framework's shopifyDataCreationHandlers slot (historical name — not Shopify-specific at runtime). It also applies squareNsSavedSearchPatch.js to strip Shopify-only NS columns from saved searches.

Every TC has a dedicated folder. Open these files to understand exactly what the test does:

FileContainsStatic or runtime?
{Test}_flowStatusJSON.jsonWhich IO flows ON/OFF (Order import, Tenders, Customer)Static
{Test}_updateSettings0.jsonIntegration settings before order (tax mode, payment mapping, lot/serial, SKU track item…)Static
{Test}_createOrder.json{ "orderKeyPrefix": "PRET16506" } — triggers handler named in testcase JSONStatic pointer; order built at runtime
{Test}_updateSettings1.jsonOn-demand order sync = {{PRET16506onDemandOrderSync}}Static template; value set at runtime
{Test}_flow_response1.jsonExpected flow job: status: completed, numSuccess, numErrorStatic
{Test}_expectedResponse1.jsonExpected NS cash sale lines, eTail fields, zero variancesStatic
{Test}_updateSettingsValidate.json(negative TCs) Settings that should return HTTP 422Static

12. Validations by System

Square (data creation)

  • Order created with correct line items, qty, modifiers, discounts, tips
  • Payment completed (Cash, Check, gift card proxy)
  • Catalog IDs resolved from env (SQUARE_CATALOG.*)
  • Order ID + on-demand key stored in test map

Integrator.io

  • Flow enabled/disabled per flowStatusJSON
  • Settings applied (updateSettings0/1)
  • Flow run accepted; job reaches completed
  • Flow response: numSuccess, numError, status
  • Negative TCs: HTTP 422 on bad on-demand settings

NetSuite (cash sale)

  • verifyCashsaleDataFromNetsuite via NS proxy
  • Line items: Item name, Quantity, Item Rate
  • eTail fields: Order Id, Channel, Currency
  • Variance fields must be 0 (tax, discount, ship, total)
  • Lookup by Square order ID (uniqueValue)

16. Demo Q&A — Common Questions

Where can I see the payload for a specific TC?
test-data/Square_Suite/Order_Import/Batch{N}/PRETxxxxx/ — 6 JSON files per TC. See Payloads section.
How is the Square order created — is it in the JSON file?
No. _createOrder.json only has orderKeyPrefix. The handler in squareDataCreation.js builds the order from a scenario at runtime and POSTs to Square Orders API.
Why is the repo called rest-api-shopify?
It's a fork/extension of the Shopify IA automation repo. Square reuses the same framework with PBI=SQNS.
What did we write vs what existed?
We wrote: Square helpers, generators, 125 testcase definitions, payload files, stability fixes, docs, env config, NS patch.
Existed: Jest framework, IO API client, NS proxy validation, test runner.
How do you run one batch?
env NODE_ENV=dev SETUP=E2E_Square SUITE=Square_Suite TAG=batch5 npm run jest
What validates pass/fail?
NS saved search result compared to _expectedResponse1.json — line items, rates, eTail variances must match.
Why batches instead of one big suite?
125 TCs grouped by feature area; run and stabilize incrementally; TAG keeps runs manageable (~15–45 min per batch).
How do I regenerate test files after editing a generator?
Run the matching node scripts/generateSquareBatchN.js then node scripts/generateSquareTeamDocs.js.
